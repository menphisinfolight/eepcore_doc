# 安全管理（.security）機制與限制

> 主題：EEP 設計畫面右鍵「安全管理」存出 `<表單>.security` 檔的機制、套用流程，以及「server 端真擋」vs「client 端視覺擋」的關鍵區分。
> 重點：**敏感欄位不可只靠 `.security`**——資料仍會送到瀏覽器。

## 操作流程

設計畫面（`/design`）→ 右鍵任一 RWD 表單 → **安全管理** → 開啟對話框 → 設定後存檔 → 產生：

```
design/bootstrap/{方案名}/{表單名}.security
```

與該表單的 `.json` 同層，屬**客製資料**（升級不會被覆蓋）。

## 對話框結構

| 區塊 | 說明 |
|------|------|
| 西側（220px） | 三個 Tab：**使用者 / 群組 / 流程活動** |
| 中央 Grid | 列出每個控制項可受控的屬性（`hidden` / `disabled` / `readonly`…）|
| 模式下拉 | `both` / `either` / `user` 三種衝突解決邏輯 |

### 三種 mode 的差異

| mode | 邏輯 |
|------|------|
| `both` | 使用者層 + 群組層**都套用**（取交集／AND）|
| `either` | 使用者或群組任一設定即生效（OR、會 toggle 屬性）|
| `user` | 使用者層**蓋過**群組層（個別設定優先）|

實作位置：`EEPBase.Core/Utility/ControlHelper.cs::SetControlValue` (L411-456)

```csharp
// both / user → 直接 return，不動屬性（保留原合併結果）
// either     → 對布林屬性做 toggle
// 其他       → control[propName] = value（直接寫入 JSON）
```

## `.security` 檔格式

JSON 格式，UTF-8 編碼，由 `MenuProvider.SaveSecurityMode` (L1715) / `SaveSecurityValue` (L1528) 透過 `StreamWriter` 寫入。

```json
{
  "users": {
    "USER_ID": [
      { "name": "control[id=txtSalary].hidden", "value": true }
    ]
  },
  "groups": {
    "GROUP_ID": [
      { "name": "control[id=btnDelete].disabled", "value": true }
    ]
  },
  "activities": {
    "START": [
      { "name": "datagrid[id=dgDetail].column[field=主管評分].hidden", "value": true }
    ]
  },
  "mode": "both"
}
```

## RWD Runtime 套用流程

`EEPRWDTools.Core/RWDPage.cs::LoadSecurity(JArray)` (L233-327)

1. 表單載入時讀 `.security`（與 `.json` 同目錄）
2. 依 `ClientInfo.Groups` 取得使用者所屬群組
3. **先合併使用者層、再合併群組層**，依 `mode` 解決衝突
4. 流程表單再加上活動層（`activities` 的 key 用 **MD5 hash** 比對活動名）
5. 透過 `ControlHelper.SetControlValues(..., MergeProperty.Security)` 把屬性合進控制項定義
6. 在送出 HTML 前完成套用

> 官方確認（[#474801](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=474801)）：「安全設定的內容在表單載入前就會進行作動」。

## ⚠️ 關鍵：哪些是 Server 端真擋、哪些只是 Client 端隱藏

`.security` 的「hidden」**不是統一的安全機制**，依控制項類型有差異：

| 控制項 / 屬性 | 實際機制 | 原始碼 / Network 能看到? | 安全等級 |
|---|---|---|---|
| **Datagrid 欄位 `hidden`** | server 把 `hidden:true` 寫進 column JSON，client 端 CSS `display:none` | ✅ **看得到**（rowData 整列下發）| ❌ 純視覺 |
| **Form 欄位 `hidden`** | server 渲染成 `<input type="hidden">` | ⚠️ DOM 有，肉眼看不見但能讀 | ⚠️ 偽裝級 |
| **`disabled` / `readonly`** | server 設屬性 → client 套上 disabled/readonly attr | ✅ DOM 在、可改 attribute 後重啟編輯 | ⚠️ 防呆級 |
| **整個 Panel / Tab** | **不支援**（[#474801](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=474801) 官方確認） | — | — |

### 為什麼 Datagrid 欄位 `hidden` 是 client 端隱藏（證據鏈）

1. `EEPRWDTools.Core/Controls/DataGrid.cs:397`
   ```csharp
   Columns.ForEach(c => c.Render(html));   // 全部欄位都 render，沒過濾
   ```
2. `Column` 類別有 class-level `[DataOption]`（預設 include all）；`Hidden` 屬性掛 `[Security("field")]` 但**沒有** `[DataOption(false)]`
3. `EEPBase.Core/Extension/ObjectExtensions.cs::GetDataOptions` (L106-200)：序列化時會把 `hidden` 屬性整個寫進 client 用的 data-options JSON
4. Client 端 `wwwroot/js/easyui/jquery.easyui.datagridview.js:89`
   ```javascript
   var style = col.hidden ? 'style="display:none;..."' : '...';
   ```
5. row 渲染照樣執行 `var value = rowData[field]` —— **資料整包送到瀏覽器**

### 安全結論

| 場景 | 適合用 `.security`? |
|------|---------------------|
| 防呆（避免使用者誤改 / 誤填）| ✅ 適合 |
| UI 簡化（不同角色看到不同操作）| ✅ 適合 |
| 非機敏欄位的可見度控制 | ✅ 適合 |
| **薪資、成本、個資等敏感欄位** | ❌ **絕對不要** |
| **角色不該看到 = 不能下發到瀏覽器** | ❌ 用 `.security` 擋不了 |

### 真正擋敏感資料的做法

| 做法 | 說明 |
|------|------|
| **DataAdapter 端過濾** | 在 SQL/查詢層依角色把欄位剔除（最徹底）|
| **ServerMethod 包一層** | 取資料前後檢查角色，剔除敏感欄位再回傳 |
| **不同 dataset / 不同表單** | 敏感角色看的是另一份去敏化 dataset |
| **欄位加密** | DB 存密文，授權角色拿到 key 才能解 |

## 流程活動層的設定

`activities` 的 key 是**活動 Title（節點名稱）**，例如「負責人主管審核」、「START」。

實務確認（[#481926](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=481926)）：要輸入活動的 Title 名稱，不是 ID。

> 注意：原始碼裡 activities key 經過 MD5 hash 比對，所以**改活動 Title 會讓既有 `.security` 設定失效**。

## 常見陷阱與限制

### 1. 群組 + 使用者會「一併參考」造成新刪修權限互相干擾（[#480502](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=480502)）

如果群組設了「新增/編輯/刪除」開啟，**個別用戶的同類設定不會生效**。

> Roland 建議：「群組/用戶擇一設定，避免互相影響。」

### 2. Container 元件（Panel / Tabs）不能設安全管理（[#474801](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=474801)）

只能對欄位級控制項設定。要隱藏整個 Panel 必須：
- 程式碼控制：`$('#Panel1').hide()`
- 或對 Panel 內**所有欄位**逐一設 hidden

### 3. 條件式權限（依資料狀態）不支援

例：「Status = 'Y' 時 G1 群組欄位 A readonly」這種動態條件**辦不到**，必須用程式碼。

### 4. 細部設定存在檔案而非 DB（[#473354](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=473354)）

使用者 / 群組對選單的權限是 DB，但欄位級的 `.security` 是檔案。

> 客戶反映「會造成維護困難」，官方規劃即如此。多站點部署要記得**同步整個 `design/bootstrap/{方案}/`** 目錄。

### 5. 選單級 vs 表單級的權限是兩套機制

| 層級 | 控制範圍 | 儲存位置 |
|------|---------|---------|
| 選單級 | 哪些群組/用戶看得到此選單 | DB |
| **表單級**（`.security`）| 表單內欄位 / 按鈕 / datagrid 欄位的 hidden / disabled / readonly | **檔案** |

### 6. AD 整合的角色設定

`.security` 用的群組 / 使用者來自 USERS / GROUPS，這些表可從 `/design` 的「**安全管理 → 匯入 AD 帳號**」匯入；匯入後 `.security` 就能直接控制 AD 群組（[#473354](https://www.infolight.com/cloud_andyhome2_bootstrap/DISDT?type=010&id=473354)）。

## 部署 / 同步注意事項

| 動作 | 影響 |
|------|------|
| 升級 EEP 版本 | `.security` 不會被覆蓋（屬 `design/bootstrap/`） |
| 多站佈署 / 客戶端佈署 | 要把整個 `design/bootstrap/{方案}/` 一起佈，含所有 `.security` |
| 改活動 Title | 既有 `activities` 設定會失效（MD5 對不上）|
| 改控制項 ID / 欄位名 | 既有設定會失效（用 `control[id=xx].property` 對應）|

## 推薦一句話

**`.security` 是「角色 UI 客製化」工具，不是「資料保護」工具。敏感欄位請在 DataAdapter / ServerMethod 端過濾，不要靠 `.security` 隱藏。**
