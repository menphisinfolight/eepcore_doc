# EEP 原始碼版本管理機制

> EEP 內建的「元件存檔歷史 + 回滾」機制。針對設計端 7 種元件（Server / Menu / Bootstrap / Client / Netflow / Processor / Report），每次存檔會把舊版本寫入 `MENUCHECKLOG` 表，之後可在設計端版本管理面板預覽、回滾。

## TL;DR

1. **開關**：`design/config/global.cfg` 的 `versionControl = "true"` 才啟用，關掉整個機制停擺
2. **觸發點**：`MenuProvider.SaveMenu` / `AddMenu` 在寫檔前呼叫 `VersionControlProvider.Submit`
3. **儲存位置**：`MENUCHECKLOG` 表的 `FILECONTENT` 欄（JSON 字串：controls + script）
4. **不是 Git、不記 diff**：只存整份快照，不同才寫一筆（Compare 判斷）
5. **Rollback 會 backup**：回滾前先把當前版本存成新記錄，然後才覆寫磁碟
6. **只保護設計端元件檔**：不保護 `design/server/*.Core.dll` 編譯結果、不保護 `SYS_*` 資料表資料

## 啟用條件

```
design/config/global.cfg  →  "versionControl": "true"
```

- **沒設或設 false**：
  - `Submit` 完全不寫 MENUCHECKLOG（但磁碟覆寫還是會發生）
  - `Rollback` 會丟 `EEPException("Versiondisabled")`
- **設 true**：
  - 存檔時自動比對並記錄
  - 版本管理面板可用

## 支援的元件類型

| FILETYPE | 對應磁碟資料夾 | 主要檔案 |
|----------|--------------|---------|
| `server` | `design/server/{Solution}/` | `{id}.json` + `{id}.cs` + 編譯出的 `{id}.Core.dll` |
| `menu` | `design/menu/{Solution}/` | `{id}.json` + `{id}.js` |
| `bootstrap` | `design/bootstrap/{Solution}/` | `{id}.json` + `{id}.js` |
| `client` | `design/client/{Solution}/` | `{id}.json` + `{id}.js` |
| `netflow` | `design/netflow/{Solution}/` | `{id}.json` + `{id}.js` + `{id}.xml` |
| `processor` | `design/processor/{Solution}/` | `{id}.json` + `{id}.js` |
| `report` | `design/report/{Solution}/` | `{id}.json` + `{id}.cs` |

> 其他類型（例如 schedule、refval 等設定類）**不經過 MenuProvider.SaveMenu**，所以不會被版本管理覆蓋。

## 核心 API

### `POST /design/` with `type=versionControl`

Provider：`EEPGlobal.Core/Provider/VersionControlProvider.cs`

| `mode` | 參數 | 功能 |
|--------|------|------|
| `load` | `id`, `menuType` | 列出某元件所有版本（對應 `menuLogView` InfoCommand）|
| `get` | `id`, `menuType`, `version` | 取版本內容。`version=0` 是特殊值 → 讀**當前磁碟**檔案（非 MENUCHECKLOG）；其他值 → 從 MENUCHECKLOG WHERE LOGID=@version 讀 |
| `editDesc` | `version`, `desc` | 改某版本的 `PACKAGE` 欄位（備註）|
| `remove` | `rows` (JArray) | 刪除版本記錄 |
| `rollback` | `id`, `menuType`, `version` | 回滾到指定版本 |

### `VersionControlProvider.Submit(id, type, datas)`

內部被 `MenuProvider` 的存檔流程呼叫，**不是對外 API**。

```
1. 讀 global.cfg，若 versionControl != "true" → no-op
2. GetVersion(id, type, "0")：讀磁碟上的「當前版本」（還沒被覆寫前的舊版本）
   → 組成 oDatas = { controls: {...}, script: "...", mtime: "..." }
3. Compare(oDatas, datas)：
   - JsonConvert.SerializeObject(data["controls"]) 字串比對
   - data["script"] 字串比對
   - 兩者都相同 → 跳過（不動作）
4. 有差異 → 把 oDatas（舊版）切成 3900 字元段，用 " to_clob" 連接
   → INSERT INTO MENUCHECKLOG (ITEMTYPE, PACKAGE, PACKAGEDATE, FILETYPE,
                               FILENAME, FILEDATE, FILECONTENT, USERID)
```

### Rollback 的特別行為

**回滾會自動 backup 當前版本**（見 `VersionControlProvider.Rollback`）：

```
Rollback(id, type, version)
  ├── 讀 MENUCHECKLOG 取舊版 FILECONTENT
  ├── Submit(id, type, { desc: "rollback backup" })
  │     → 當前磁碟版本先被寫成新 MENUCHECKLOG 記錄
  ├── 舊版 controls 覆寫 {id}.json
  ├── 舊版 script  覆寫 {id}.js
  └── type == netflow 時額外 Flow.SaveToXml 重建 .xml
```

這表示回滾不會遺失當前狀態 —— 下次想要「再滾回來」時，可以從版本列表找到那筆 `"rollback backup"`。

## Compare 的判斷依據

```csharp
// VersionControlProvider.cs L204-213
public bool Compare(JObject data1, JObject data2)
{
    var controls1 = JsonConvert.SerializeObject(data1["controls"]);
    var controls2 = JsonConvert.SerializeObject(data2["controls"]);
    var script1 = data1["script"] == null ? "" : data1["script"];
    var script2 = data2["script"] == null ? "" : data2["script"];
    return controls1 == controls2 && script1 == script2;
}
```

- 只比 `controls`（元件設計結構）+ `script`（C# / JS 原始碼）
- `controls` 用 `JsonConvert.SerializeObject` 序列化比字串 —— **property 順序不同會被當成不同**（但 save 流程重序列化後通常穩定）
- **不比 `mtime`、`xml`（netflow）**：就算 xml 不同但 controls 一致也不會寫新版本

## 為什麼要切 3900 字元段？

```csharp
var fCount = (int)Math.Ceiling(1.0 * oDatas.ToString().Length / 3900);
var fileContentArr = new List<string>();
for (int i = 0; i < fCount; i++) {
    var fn = oDatas.ToString().Skip(i * 3900).Take(3900).ToArray();
    fileContentArr.Add(new string(fn));
}
var newFileContent = String.Join(" to_clob", fileContentArr);
```

- Oracle 字串字面值上限 **4000 字元**（`VARCHAR2`），超過要改用 `TO_CLOB() || TO_CLOB() || ...` 拼接才能塞進 CLOB 欄位
- 程式一律做 3900 切段 + ` to_clob` 連接，不分資料庫
- **讀回時 GetVersion 會 `content.Replace(" to_clob", "")`** 把分段標記拿掉還原成單一字串
- 對 SQL Server / MySQL / DB2 / Informix（儲存型別為 image / LONGBLOB / BLOB）沒有負面影響，但 **byte[] 型態**要額外用 `Encoding.UTF8.GetString((byte[])rows[0]["FILECONTENT"])` 才能解回字串（見 `GetVersion` L118-122）

## MENUCHECKLOG 表摘要

| 關鍵欄位 | 語意 |
|---------|------|
| `LOGID` | PK（版本識別碼，前端 version 參數傳這個）|
| `ITEMTYPE` | 方案隔離（`ClientInfo.Solution`）|
| `PACKAGE` | 版本描述；常見：`"overwrite backup"` / `"rollback backup"` / 使用者輸入 |
| `PACKAGEDATE` | 版本記錄建立時間 |
| `FILETYPE` | `server` / `menu` / `bootstrap` / `client` / `netflow` / `processor` / `report` |
| `FILENAME` | 元件 id |
| `FILEDATE` | **舊檔案的原始 mtime**（不是版本建立時間）|
| `FILECONTENT` | 舊版 JSON 快照（controls + script，分段 ` to_clob` 連接）|
| `USERID` | 觸發存檔的使用者（僅 SQL Server 有此欄）|

👉 完整欄位、跨資料庫型別、ALTER 腳本見系統資料表文件 **MENUCHECKLOG**（側邊欄：系統資料表 → 選單系統 → MENUCHECKLOG）

## 限制與已知問題

| 問題 | 說明 |
|------|------|
| **不是真的 Git** | 只存整份快照，沒有 branch、merge、diff（UI 層可能有顯示差異，但後端是整檔比對）|
| **只比 controls + script** | netflow 的 xml 檔不參與 Compare，但 Rollback 時會重建 xml（從 controls）|
| **CLOB 字面值切段只支援 Oracle 原意** | 其他 DB 雖然不需要，但也用同一套切法，沒副作用但多了 `" to_clob"` 字串處理 |
| **USERID 跨 DB 不一致** | 只有 SQL Server 有此欄位（還是 ALTER 補上的），其他 DB 的建表腳本沒這欄 —— Oracle/MySQL/DB2/Informix 看不到操作者 |
| **不保護 DLL** | `{id}.Core.dll` 是 MSBuild 編譯出的產物，Rollback 只還原 `.cs`，DLL 需要下次存檔才重新編譯；若 Server 還在跑會有「程式和 DLL 不一致」的短暫窗口 |
| **不保護元件之外的東西** | `design/config/`、`SYS_*` 系統資料表資料、schedule / refval 設定都不在保護範圍 |
| **無過期清理** | 表會一直長，需手動清理（版本管理面板 `remove` 或直接 SQL DELETE）|

## 常見操作

### 停用版本管理

```
# design/config/global.cfg
"versionControl": "false"
```

會讓 Submit 直接 no-op、Rollback 丟 `Versiondisabled` 例外。

### 手動清老舊版本

在設計端版本管理面板勾選 → 刪除，或直接：

```sql
DELETE FROM MENUCHECKLOG
WHERE PACKAGEDATE < DATEADD(month, -6, GETDATE())   -- 半年前以前
  AND FILETYPE = 'server'
  AND ITEMTYPE = 'SOLUTION1';
```

### 查某元件的版本歷史

```sql
SELECT LOGID, PACKAGE, PACKAGEDATE, FILEDATE, USERID
FROM MENUCHECKLOG
WHERE FILETYPE = 'server'
  AND FILENAME = 'YourModuleName'
  AND ITEMTYPE = 'YourSolution'
ORDER BY PACKAGEDATE DESC;
```
